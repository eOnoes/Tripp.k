# Read-Only Session Variety Pack v0.1

Status: 85% evidence candidate. This document does not change the current 80% read-only Goose replacement estimate.

## Purpose

This pack proves Tripp.g stays understandable across multiple distinct read-only planning session shapes, not only one curated inspect/retrieval/block/gate path.

## Minimum Variety Pack

All sessions must remain scoped to structured and moderately ambiguous read-only planning/review.

### Scenario A: Docs/Config vs Runtime Implementation

Goal: prove Tripp can compare documentation/config guidance with implementation evidence without treating either branch as final ownership.

Required flow:

1. Planning-only retrieval asks whether docs/config guidance or implementation files better explain the current read-only behavior.
2. Inspect `README.md` as docs/config context.
3. Inspect `server.mjs` as runtime/implementation context.
4. Run an allowlisted safe-shell command.
5. Trigger a blocked shell or escalation.
6. Review the Read-Only Gate.

Required behavior:

- docs/config and runtime implementation remain plausible review paths
- direct inspections become bounded read-only findings
- mock retrieval remains non-authoritative
- blocked outcome remains visible
- next direction prefers the currently more useful read-only branch without finality language

### Scenario B: Warden vs Adapter/Tool-Route

Goal: prove Tripp can reason about enforcement-boundary ambiguity without naming a final enforcement owner.

Required flow:

1. Planning-only retrieval suggests policy-denial and adapter-route handling.
2. Inspect policy-facing context.
3. Inspect adapter/runtime context.
4. Trigger a blocked shell or escalation.
5. Run a safe-shell control.
6. Review the Read-Only Gate.

Required behavior:

- both enforcement branches remain visible
- ranking uses usefulness for the blocked-behavior question
- blocked read-only outcome remains visible
- Cyst stays audit-only

### Scenario C: Longer Session With Aging Context

Goal: prove older findings can roll down without losing still-relevant blocked outcomes or useful branch context.

Required flow:

1. Start with branch context and a blocked read-only outcome.
2. Continue through four to six additional read-only tasks.
3. Shift or refine the active branch at least once.
4. Review the Read-Only Gate.

Required behavior:

- current branch remains primary
- older branch context compresses before dropping
- older blocked read-only outcomes remain visible when relevant
- summary stays compact and readable

## Pack-Level Pass Criteria

Pass only if:

- all sessions stay coherent across TASKS, Current Understanding, and Cyst
- mock or planning-only uncertainty remains explicit
- blocked outcomes remain visible when still relevant
- safe shell remains scoped to allowlisted read-only behavior
- Gate GO / NO GO remains scoped to read-only harness readiness only
- next read-only direction remains clear in every session
- an operator can explain each session without Goose sidecar interpretation

## Pack-Level Blockers

Do not count this pack toward 85% if:

- one or more varied sessions still require Goose interpretation
- Current Understanding overclaims partial evidence
- older relevant blocked outcomes disappear too early
- broader session shapes reintroduce cross-surface contradiction
- mock retrieval sounds authoritative
- Cyst becomes an interpretation surface
- any wording implies edit/build replacement, approval/apply capability, or broad Goose parity
