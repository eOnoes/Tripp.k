# Post-85 Read-Only Roadmap v0.1

Status: roadmap contract. This document does not enable runtime writes or change the current scoped 85% read-only planning/review readiness estimate.

## Current Scope

Tripp.g is at 85% for structured and moderately ambiguous read-only planning/review only.

This does not include:

- edit/build replacement
- live writes
- approval/apply runtime behavior
- broad Goose parity
- open-ended engineering reasoning parity
- runtime mutation of any kind

## Next Milestone Target

90% Read-Only Planning/Review Readiness.

This milestone remains scoped to read-only planning/review. It should prove stronger generalization across broader everyday read-only sessions, not a new write-capable capability class.

## Required Proof For 90%

1. Broader read-only breadth
   - more scenario families
   - more repo/question shapes
   - less curated session structure
   - varied docs/runtime/policy/adapter combinations

2. Deeper synthesis quality
   - better summaries under weak, mixed, partial, or aging evidence
   - better uncertainty quality
   - safer compression of older context
   - mixed-signal summaries that stay useful without sounding settled

3. Long-session stress
   - 8 to 12+ read-only tasks
   - multiple branch shifts
   - multiple blocked outcomes
   - persistent cross-surface coherence
   - operator can reconstruct the session without Goose interpretation

4. Beta QA and release discipline
   - no wording drift
   - no cross-surface contradiction
   - stable operator-independence evidence across broader packs
   - release wording remains scoped to read-only planning/review

5. Future write design remains docs-only
   - no runtime behavior change
   - no UI shift toward mutation flows
   - no current write readiness claim

## Recommended Trains

### Train 1: Broader Read-Only Breadth

- expand the scenario pack
- cover more distinct question types
- cover more docs/runtime/policy/adapter combinations
- include more mixed blocked/allowed flows

### Train 2: Deeper Synthesis Quality

- improve partial-evidence phrasing
- strengthen uncertainty quality
- improve aging-context compression
- add mixed weak-signal summarization proof

### Train 3: Long-Session Stress

- add longer sessions
- add repeated branch shifts
- add multiple blocked outcomes
- add continuity stress cases

### Train 4: Beta QA Discipline

- harden release wording
- harden verifier consistency
- harden pack artifact integrity
- harden cross-surface drift checks

### Train 5: Future Write Design Only

- keep write lifecycle work as docs/contracts only
- do not enable runtime mutation
- do not introduce approval/apply UI

## 90% Blockers

Do not move toward 90% if:

- broader session shapes still require Goose interpretation
- Current Understanding overclaims under partial or aging evidence
- older blocked outcomes disappear too aggressively
- long sessions become hard to reconstruct
- score or docs start sounding like broad Goose replacement
- future write docs blur current read-only scope

## Claim Guardrails

- Always pair 85% and any future 90% language with structured/moderately ambiguous read-only planning/review scope.
- Keep edit/build replacement as a separate milestone.
- Keep future write lifecycle design explicitly non-operative.
- Keep Cyst as audit/timeline truth, not interpretation.
- Keep TASKS and Current Understanding as interpretation surfaces.
